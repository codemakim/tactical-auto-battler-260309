# 영입 상점 스펙 (Recruit Shop Specification)

> Source of Truth for Town의 신규 멤버 영입 루프.
> 기능 구현 전 기준 문서로 사용한다.

---

## §1. 목적

현재 메타 루프에는 `새 캐릭터 획득 수단`이 부족하다.
스타터 4명 이후 로스터를 늘릴 수단이 없으면:
- 훈련소의 장기 성장 의미가 약해지고
- 편성/프리셋 다양성이 줄고
- 플레이어가 골드를 어디에 써야 하는지 애매해진다

이 스펙의 목적은 Town 상점에서 `골드로 신규 멤버를 영입`하는 MVP 루프를 정의하는 것이다.

---

## §2. 기본 개념

### §2.1 상점 역할

Town의 `상점`은 액션 카드 상점이 아니다.
현재 MVP에서는 **신규 멤버 모집 상점**으로 사용한다.

즉 상점의 핵심 기능은:
- 영입 후보 3명 노출
- 골드로 즉시 영입
- 목록 리프레시

### §2.2 후보 슬롯

- 상점은 항상 `후보 3명`을 가진다
- 후보는 현재 로스터에 없는 신규 캐릭터 정의여야 한다
- 한 슬롯의 후보를 사면 그 슬롯은 `빈 슬롯`이 된다
- 자동 개별 보충은 하지 않는다
- 새 후보를 보려면 `리프레시`를 사용해야 한다

이유:
- 구현이 단순하다
- 플레이어가 무엇을 샀고 무엇을 넘겼는지 명확하다
- 리프레시 비용의 의미가 분명해진다

## §3. 상점 UI

### §3.1 위치

- Town에서 `상점` 건물 클릭
- 전용 오버레이 패널 표시

### §3.2 표시 항목

후보 카드마다 최소 표시:
- 이름
- 클래스
- 핵심 스탯 요약 (`HP / ATK / GRD / AGI`)
- 가격

상단 또는 하단 공통 영역:
- 현재 골드
- `REFRESH` 버튼
- 리프레시 비용

### §3.3 버튼

- 각 후보 카드: `영입`
- 공통: `REFRESH`
- 공통: `닫기`

---

## §4. 가격 정책

### §4.1 영입 가격

MVP에서는 모든 후보를 동일 가격으로 둔다.

- 추천값: `80 Gold`

이유:
- 초기 밸런싱 단순화
- 클래스별 가격 차등은 나중에 붙여도 됨

### §4.2 리프레시 가격

- 추천값: `20 Gold`

이유:
- 무료 리롤 악용 방지
- 하지만 지나치게 비싸지 않아야 함

---

## §5. 후보 생성 규칙

### §5.1 기본 규칙

후보는 아래 조건을 만족해야 한다.

1. 현재 로스터에 없는 캐릭터
2. 현재 상점 후보와 중복되지 않는 캐릭터
3. 클래스/이름/초기 스탯이 고정된 `CharacterDefinition` 기반

### §5.2 초기 후보 생성

- 새 게임 시작 시 3명 생성
- 세이브 로드 시 저장된 후보 유지

### §5.3 수동 리프레시

- 골드가 충분할 때만 가능
- 비용 차감 후 전체 3슬롯을 새 후보로 교체
- 이미 구매해서 비어 있는 슬롯도 같이 새 후보로 채움

---

## §6. 자동 갱신 규칙

이 부분이 가장 중요하다.

### §6.1 자동 갱신 트리거

상점 후보는 아래 경우에만 자동 갱신된다.

- **1스테이지 이상 클리어한 런이 종료되었을 때**

즉 다음은 자동 갱신된다.
- 첫 스테이지 이상 클리어 후 패배/포기
- 런 완주

다음은 자동 갱신되지 않는다.
- 런 시작 직후 바로 포기
- 첫 전투 전에 포기
- 스테이지 1 클리어 전에 런 종료

### §6.2 기준 필드

자동 갱신 판단은 `RunResultData.stagesCleared >= 1` 또는 동등한 런 종료 시점 정보로 판단한다.

### §6.3 이유

이 규칙을 쓰는 이유:
- “런 시작 → 바로 포기”로 후보 리롤 악용 방지
- 실제 플레이를 유도
- 규칙이 이해하기 쉽고 일관적

---

## §7. 영입 처리

### §7.1 성공 시

1. 골드 차감
2. 캐릭터를 `gameState.characters`에 추가
3. 해당 후보 슬롯 비우기
4. 저장

### §7.2 실패 조건

아래 경우 영입 불가:
- 골드 부족
- 캐릭터 슬롯 부족 (`characters.length >= maxCharacterSlots`)
- 후보 슬롯이 이미 비어 있음

### §7.3 슬롯 부족 처리

MVP에서는 영입 버튼 비활성 + 안내 문구만 표시한다.

예:
- `로스터가 가득 찼습니다`

슬롯 확장 구매는 이번 작업 범위 밖이다.

---

## §8. 세이브 데이터

상점은 저장 대상이다.

저장해야 하는 것:
- 현재 영입 후보 3명

저장 이유:
- 새로고침으로 상점이 바뀌면 안 됨
- 플레이어가 Town을 나갔다 돌아와도 같은 후보를 봐야 함
- 타이틀 복귀/세이브 로드 후에도 상태가 유지되어야 함

## §9. 데이터 모델 초안

```typescript
interface RecruitShopOffer {
  slotIndex: number;
  character: CharacterDefinition | null;
  price: number;
}

interface RecruitShopState {
  offers: RecruitShopOffer[];
  refreshCost: number;
}
```

MVP에서는 이 정도면 충분하다.

## §10. 현재 설계 결정

이번 스펙에서 확정한 것:

1. Town 상점은 신규 멤버 영입 상점으로 사용
2. 후보는 3명
3. 수동 리프레시는 골드 소모
4. 영입 후 슬롯은 비우고 자동 보충 안 함
5. 자동 갱신은 `1스테이지 이상 클리어한 런 종료`일 때만
6. 출격만 하고 포기하는 식의 리롤은 허용하지 않음

## §11. 이번 작업에서 제외

- 클래스별 가격 차등
- 희귀도 시스템
- 상점 등급/티어
- 슬롯 확장 구매
- 캐릭터 판매/해고
- 런 중 상점
- 전장 클리어 보상 캐릭터

## §12. 구현 순서 제안

1. `RecruitShopState` 저장 구조 추가
2. 후보 생성/리프레시 순수 함수 추가
3. 자동 갱신 트리거를 런 종료 정산 시 연결
4. Town 상점 오버레이 구현
5. 저장/로드 테스트
6. 영입/리프레시/자동 갱신 테스트

# 편성 화면 재설계 계획

> 상태: 구현 완료 (Codex, 2026-03-30)

> 목적: 현재 `FormationScene`에 과도하게 모여 있는 기능을 분리해
> 메인 화면은 "배치"에 집중시키고, 세부 편집은 오버레이로 분리한다.

---

## 1. 문제 정의

현재 편성 화면은 한 장에서 아래 기능을 모두 처리한다.

- 캐릭터 선택
- FRONT / BACK 배치
- 영웅 선택
- 캐릭터 행동 카드 편집
- 프리셋 저장 / 불러오기 / 삭제

이 구조 때문에:

- 메인 목적이 무엇인지 흐려진다
- 하단 패널이 과밀해진다
- 우측 상세 패널이 카드 편집까지 떠안아 밀도가 과해진다
- 캐릭터 배치 존에서 텍스트 위주 표현이라 시각적 만족감이 낮다

---

## 2. 재설계 목표

### 핵심 원칙

- 메인 화면은 `배치`를 최우선으로 보여준다
- 세부 작업은 오버레이로 분리한다
- 편성 존에는 캐릭터 이름만이 아니라 실제 캐릭터 스프라이트를 표시한다
- 카드 편집과 프리셋 관리는 메인 레이아웃을 오염시키지 않는다
- 영웅 선택도 상시 패널이 아니라 필요 시 여는 `COMMAND` 오버레이로 다룬다

### 메인 UX 목표

플레이어가 편성 화면에 들어오면 가장 먼저 보이고 다뤄야 하는 것은:

1. 누구를 파티에 넣을지
2. 어디에 배치할지
3. 어떤 영웅을 쓸지

---

## 3. 목표 구조

## 3.1 메인 편성 화면

메인 화면에는 아래만 남긴다.

- 좌측: 보유 캐릭터 목록
- 중앙: BACK / FRONT 배치 존
- 중앙 하단: 현재 영웅 + 선택 유닛 요약 HUD
- 하단 버튼:
  - `COMMAND`
  - `PRESETS`
  - `TACTICS`
  - `뒤로`
  - `DEPLOY`

### 제거 대상

메인 화면에서 제거:

- 프리셋 패널 상시 표시
- 카드 슬롯 3장 + 인벤토리 상시 표시
- 영웅 선택 상시 패널
- 선택 캐릭터 우측 상세 패널

즉, 메인 화면은 `ROSTER + BOARD + 하단 액션` 중심으로 정리한다.

---

## 3.2 오버레이 분리

### A. 카드 편집 오버레이

역할:

- 선택 캐릭터의 액션 슬롯 확인
- 카드 장착 / 해제
- 슬롯 순서 변경
- 런 중 인벤토리 카드 장착

구성:

- 상단: 캐릭터 이름 / 클래스 / 기본 스탯
- 중단: 슬롯 카드 3장
- 슬롯 사이: 스왑 버튼
- 하단: 인벤토리 카드 목록
- 닫기 버튼

호출 조건:

- 메인 화면에서 캐릭터가 선택된 상태에서 `카드 편집` 버튼 클릭

캐릭터 미선택 시:

- 토스트 또는 모달로 "캐릭터를 먼저 선택하세요"

### B. 프리셋 오버레이

역할:

- `Preset 1~3` 저장 / 불러오기 / 삭제

구성:

- 상단: 현재 선택 슬롯
- 중단: Preset 1, 2, 3
- 하단: 저장 / 불러오기 / 삭제
- 현재 프리셋 요약 텍스트

메인 화면에서는 버튼만 두고, 실제 조작은 오버레이에서 한다.

### C. COMMAND 오버레이

역할:

- 영웅 선택
- 현재 영웅 특화 능력 요약
- 런 중 영웅 변경 잠금 상태 안내

메인 화면에서는 현재 영웅 이름만 보드 하단 HUD에 노출하고,
실제 선택은 오버레이에서 한다.

---

## 4. 편성 존 캐릭터 표현 변경

## 4.1 현재 문제

현재 중앙 존에는 캐릭터 이름과 클래스 약어 중심으로 표시된다.

문제:

- 시각적으로 밋밋하다
- 배치 만족감이 약하다
- 캐릭터가 "유닛"이 아니라 "텍스트 박스"처럼 보인다

## 4.2 목표

전투 화면에서 쓰는 캐릭터 스프라이트를 축소 재사용해,
편성 존에서도 각 캐릭터가 실제 모습으로 보이게 한다.

## 4.3 사용할 자산

현재 확인된 자산:

- `warrior-attack.png`
- `assassin-attack.png`
- `archer-attack.png`
- `guardian-attack.png`
- `controller-attack.png`

전투 화면 기준 idle frame:

- `WARRIOR` → `warrior-attack`, frame `0`
- `ASSASSIN` → `assassin-attack`, frame `0`
- `ARCHER` → `archer-attack`, frame `27`
- `GUARDIAN` → `guardian-attack`, frame `0`
- `CONTROLLER` → `controller-attack`, frame `0`

즉 편성 화면도 동일 규칙을 따라
"공격 스프라이트시트의 idle frame"을 정적 표시로 사용한다.

## 4.4 렌더링 원칙

각 존 유닛은 아래 요소로 구성:

- 캐릭터 스프라이트
- 이름
- 클래스 약어
- 짧은 스탯 요약 또는 선택 상태 표시
- 제거 버튼 `x`

전투 화면처럼 과한 연출은 넣지 않는다.
정적 idle 표시만 사용한다.

## 4.5 스케일 방향

전투 화면보다 축소된 카드형 유닛 비주얼로 표시한다.

추천:

- 스프라이트 중심
- 이름은 하단
- 선택 시 외곽선 강조

텍스트 박스를 스프라이트 위에 덧씌우는 방식보다,
"스프라이트 + 이름 라벨" 구조가 더 자연스럽다.

### 4.6 존 레이아웃 보정

- 세로 박스 2개는 스프라이트 도입 후 밀도가 과해지므로 사용하지 않는다
- `BACK / FRONT`는 가로 스트립 2줄로 재배치한다
- 각 스트립은 한 줄에 최대 5슬롯까지 수용 가능한 폭으로 잡는다
- 현재 전투 출전 인원 제한은 4명이므로, 5번째 슬롯은 미래 확장 여유로 본다

---

## 5. 선택 유닛 정보 재정의

상세 패널을 메인 화면에서 제거하고,
선택 유닛 정보는 보드 하단 HUD로 축약한다.

표시 항목:

- 이름
- 클래스
- 핵심 스탯
- 현재 라인
- 행동 카드 이름 요약

상세 카드 편집은 `TACTICS` 오버레이에서만 다룬다.

---

## 6. 영웅 선택 확장 대비

사용자 요구:

- 나중에는 영웅도 "캐릭터처럼 보이게" 만들 예정

따라서 지금 재설계는 영웅 선택도 향후 확장 가능한 구조로 잡는다.

### 현재 단계

- 메인 화면에는 현재 영웅 이름만 노출
- 실제 영웅 선택은 `COMMAND` 오버레이에서 처리

### 향후 확장 고려

- 영웅도 초상화/스프라이트 카드형 선택 UI로 교체 가능하도록
  배치 공간을 넉넉하게 둔다
- 메인 편성 화면에서 영웅 선택 섹션은 독립 블록으로 유지한다

즉, 카드 편집/프리셋을 빼내야
영웅 비주얼 확장 공간도 확보된다.

---

## 7. 구현 단계 제안

### Step 1. 메인 화면 정리

- 프리셋 패널 제거
- 카드 편집 UI 제거
- 우측 패널 제거
- 하단에 `COMMAND`, `PRESETS`, `TACTICS` 버튼 추가

### Step 2. 프리셋 오버레이 분리

- 현재 프리셋 기능을 메인 패널에서 제거
- 별도 오버레이로 이동

### Step 3. 카드 편집 오버레이 분리

- 현재 우측 상세 패널의 카드/인벤토리 기능을 오버레이로 이동

### Step 4. 편성 존 스프라이트 적용

- 존 유닛을 텍스트 박스가 아닌 스프라이트 카드로 표시

### Step 5. 보드 HUD 정리

- 현재 영웅 / 선택 유닛 요약을 보드 하단 HUD로 축소
- 메인 화면이 `ROSTER + BOARD` 중심으로 보이게 조정

- 클래스별 idle sprite 매핑 추가
- 존 유닛 비주얼을 텍스트 박스에서 스프라이트형으로 교체

### Step 6. 구조 리팩터링 1차

- `FormationSceneLayout.ts`로 레이아웃 좌표/오버레이 크기/스프라이트 매핑 추출
- `FormationSceneStyles.ts`로 로스터/유닛/COMMAND 카드 상태 스타일 추출
- `FormationGraphics.ts`로 반복되는 rounded frame 렌더링 헬퍼 추출
- `FormationScene`는 상태 연결과 이벤트 처리 중심으로 축소

이 단계에서는 Scene 분할까지는 하지 않고,
다음 패스에서 `Roster / Board / HUD / Overlay` 뷰 클래스로 나누는 것을 목표로 한다.

### Step 5. 마감

- 선택 상태, hover, remove 버튼 정리
- 화면 높이/폭 재조정
- 모바일/저해상도에서 겹침 확인

## 8. 비주얼 톤 가이드

- 문서형 라벨(`영웅 선택`, `선택 캐릭터`, `BACK 영역 (후열)`)은 피한다
- 편성 화면은 관리자 패널이 아니라 전술 보드처럼 보여야 한다
- 존 타이틀은 짧은 전술 용어로 유지한다
  - 예: `BACKLINE`, `FRONTLINE`, `COMMAND`, `UNIT`
- 중앙 배치 영역은 패널 여러 개보다 하나의 보드 위 2개 라인처럼 읽혀야 한다

---

## 8. 리스크

### 1. FormationScene가 이미 역할이 큼

해결:

- 오버레이 렌더링 로직을 내부 메서드로 먼저 분리
- 필요하면 이후 `FormationCardOverlay`, `FormationPresetOverlay` 같은 UI 보조 모듈로 추출

### 2. 스프라이트 비율이 편성 화면과 맞지 않을 수 있음

해결:

- 전투 화면 scale을 그대로 쓰지 말고
- 편성 전용 scale 상수를 별도로 둔다

### 3. LANCER 등 향후 클래스 자산

해결:

- 클래스별 스프라이트 매핑은 공용 테이블로 분리
- 자산 없는 클래스는 임시 fallback 허용

---

## 9. 권장 구현 순서

1. 프리셋 오버레이 분리
2. 카드 편집 오버레이 분리
3. 우측 패널 요약화
4. 편성 존 스프라이트 적용

이 순서가 좋은 이유:

- 레이아웃 복잡도를 먼저 줄인다
- 그 다음 비주얼 업그레이드를 적용한다
- 스프라이트 적용 전에 화면 구조가 안정돼야 한다

---

## 10. 이번 결정 요약

- 메인 편성 화면은 `배치 중심`
- 카드 편집은 오버레이
- 프리셋도 오버레이
- 편성 존에는 이름 대신 실제 캐릭터 스프라이트를 표시
- 스프라이트는 전투 화면 idle frame 자산을 재사용

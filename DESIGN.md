# DESIGN.md

이 문서는 게임 화면을 만들 때 따르는 시각/카피 기준이다.
목표는 “기능이 보이는 UI”가 아니라 “플레이어가 게임 안의 전술 화면으로 받아들이는 UI”다.
플랫폼/해상도 기준은 [docs/platform-ui-baseline-spec.md](/Users/jhkim/Project/tactical-auto-battler/docs/platform-ui-baseline-spec.md)를 따른다.

## Product Tone

- 어두운 전술 판타지 캠프, 작전 보고서, 병영, 전장 지도 느낌.
- 개발툴, 관리자 패널, 문서 뷰어, 스프레드시트 느낌은 피한다.
- 설명보다 배치, 색, 아이콘, 상태 대비로 읽히게 한다.

## Copy Rules

- 짧게 쓴다.
- 괄호 설명을 피한다. 예: `BACK 영역 (후열)` 금지.
- 구현 용어를 플레이어에게 노출하지 않는다.
- 삭제된 기능은 카피에 남기지 않는다. 예: 객원 멤버.
- 액션 카드 표기는 플랫 태그 나열을 기본으로 한다. `상황 / 대상 / 효과` 섹션 라벨을 만들지 않는다.
- 자기 자신은 `나`로 통일한다. 예: `나`, `나 전열`, `나 후열`.

## Layout Rules

- 한 화면에 주인공 영역 하나를 둔다.
- 모든 정보를 같은 크기의 박스로 나누지 않는다.
- 메인 화면은 현재 행동에 필요한 정보만 노출한다.
- 깊은 정보는 오버레이나 호버 상세로 보낸다.
- 패널 안의 패널을 반복하지 않는다.
- 와이드 화면에서는 게임 캔버스가 중앙에 와야 한다.

## Component Rules

- 카드, 배지, 버튼은 같은 축과 간격을 공유한다.
- 배지는 텍스트 실제 폭 + 고정 패딩으로 만든다. 상자에 텍스트를 억지로 맞추지 않는다.
- 태그가 많은 카드는 줄 수를 제한하고 `+N`으로 숨겨진 태그 수를 표시한다.
- 오버레이는 바깥 dim 영역 클릭으로 닫을 수 있지만, 오버레이 내부 빈 공간 클릭으로 닫히면 안 된다.

## Color And Emphasis

- 파랑 계열: 나/아군/조건
- 빨강 계열: 적/대상
- 보라 계열: 효과/전술
- 금색 계열: 선택, 희귀, 보상, 중요 결정
- 회색 계열: 비활성, 잠금, 보조 정보

색은 의미를 가져야 한다. 단순 장식용 색 증식은 피한다.

## Icon Rules

- 액션 효과는 전용 모노 아이콘 세트를 사용한다.
- 컬러 이모지는 기본값으로 쓰지 않는다.
- 아이콘은 의미를 돕는 보조 수단이며, 텍스트를 완전히 대체하지 않는다.
- 공격, 실드, 회복, 이동, 밀침, 버프/디버프, 턴 조작은 같은 아이콘 세트를 공유한다.

## Screen-Specific Guardrails

### Formation

- 메인 화면은 `ROSTER + TACTICAL BOARD` 중심이다.
- 상세 정보와 프리셋/카드 편집은 필요할 때만 연다.
- 전열/후열은 위치와 라인으로 이해되게 하고, 장황한 설명을 붙이지 않는다.

### Reward

- 보상 카드는 균등 간격의 플랫 카드열을 기본으로 한다.
- 선택 카드 강조는 테두리, 깊이, 톤으로 처리한다.
- 런 중 획득한 임시 액션 카드를 영구 보상처럼 표현하지 않는다.

### Town

- 마을 건물은 메타 루프의 진입점이다.
- 상점/병영/훈련소 오버레이는 필요한 정보만 보여준다.
- 반복 설명보다 결과와 비용, 실행 가능 상태를 우선한다.

### Title

- 저장 상태는 개발자 메시지가 아니라 게임 내 브리핑처럼 보여야 한다.
- 손상 저장, 활성 런, 일반 저장 상태는 카피와 색으로 구분한다.

## Anti-Patterns

- 긴 설명문으로 UI 의도를 설명하기
- 모든 영역에 테두리 박스 넣기
- 괄호로 번역/역할을 반복하기
- 영문 개발 키를 그대로 표시하기
- 삭제된 시스템을 보상/상태 카피에 남기기
- 중앙 정렬 기준이 화면마다 흔들리기
- 한 컴포넌트는 새 표현, 다른 컴포넌트는 옛 표현을 유지하기
